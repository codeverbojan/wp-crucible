# Starter Plugin — Development Guide

> AI-assistant-friendly reference for contributing to this WordPress plugin.
> Human developers: this doubles as your onboarding doc.

## Quick Start

```bash
# 1. Clone and enter the plugin directory.
git clone <repo-url> starter-plugin
cd starter-plugin

# 2. Install PHP dev dependencies (linting, testing, static analysis).
composer install

# 3. Install JS dev dependencies (webpack, linting).
npm ci            # NOT npm install — deterministic lockfile installs only.

# 4. Build front-end assets (CSS/JS → assets/build/).
npm run build

# 5. (Optional) Start a local WordPress environment.
npm run env:start # Launches wp-env with PHP 8.2 and WP_DEBUG enabled.
```

Verify everything works:

```bash
composer check    # Runs PHPCS + PHPStan + PHPUnit in sequence.
npm run lint:js   # ESLint with @wordpress/scripts config.
npm run lint:css  # Stylelint on assets/src/css/.
```

## Rename Guide

Use `bin/rename.sh` to rebrand this boilerplate for your own plugin:

```bash
# Interactive — prompts for slug and namespace:
bash bin/rename.sh

# Non-interactive:
bash bin/rename.sh my-awesome-plugin MyAwesomePlugin

# Preview changes without modifying files:
bash bin/rename.sh --dry-run my-awesome-plugin MyAwesomePlugin
```

The script derives all naming variants automatically:

| Variant | Example | Used for |
|---------|---------|----------|
| `SLUG` | `my-awesome-plugin` | Text domain, CSS handles, directory name |
| `NAMESPACE` | `MyAwesomePlugin` | PHP namespace |
| `PREFIX` | `my_awesome_plugin_` | Options, hooks, function prefixes |
| `CONSTANT_PREFIX` | `MY_AWESOME_PLUGIN_` | PHP constants |
| `GLOBAL_VAR` | `my_awesome_plugin` | `$GLOBALS` key (no trailing underscore) |

**Replacement order matters.** The script processes from most specific to least
specific to avoid partial-match corruption (e.g., replacing the slug before the
prefix would produce `my-awesome-plugin_` instead of `my_awesome_plugin_`):

```
1. STARTER_PLUGIN_  → MY_AWESOME_PLUGIN_   (constant prefix with _ — most specific)
2. STARTER_PLUGIN   → MY_AWESOME_PLUGIN    (constant prefix without _ — for PHPCS config)
3. StarterPlugin    → MyAwesomePlugin       (namespace)
4. starter_plugin_  → my_awesome_plugin_    (underscore prefix)
5. starter_plugin   → my_awesome_plugin     (global variable key)
6. starter-plugin   → my-awesome-plugin     (hyphenated slug — least specific)
7. Rename starter-plugin.php → my-awesome-plugin.php
8. Rename starter-plugin/    → my-awesome-plugin/  (if running from parent)
```

The script also updates `phpcs.xml.dist` (text domain + prefix attributes),
`composer.json` (autoload namespace), and `.wp-env.json` (plugin path).

**Manual fallback:** If the script fails, perform the 7 replacements above in
the listed order using find-and-replace across all project files. Rename the
main PHP file and directory last.

## Coding Conventions

### PHP

- **`declare( strict_types=1 );`** as the first statement in every PHP file (after the opening `<?php` tag and docblock). PHP requires this before any other executable code.
- **`defined( 'ABSPATH' ) || exit;`** guard after `declare` and `namespace` — prevents direct file access.
- **PSR-4 autoloading** — `StarterPlugin\` maps to `src/`. Class `StarterPlugin\Admin\SettingsPage` lives at `src/Admin/SettingsPage.php`.
- **WPCS file naming exclusion** — WPCS expects `class-settingspage.php` but PSR-4 requires `SettingsPage.php`. The `phpcs.xml.dist` already excludes this rule.
- **Named callbacks for hooks** — always use `[ $this, 'method_name' ]` or `[ ClassName::class, 'static_method' ]`. Never use anonymous closures for hooks because they cannot be unhooked by other plugins or tests.
- **Global instance** — the plugin instance lives in `$GLOBALS['starter_plugin']`, set by the main plugin file. No service container, no singleton accessor — just a plain global. Access it via `$GLOBALS['starter_plugin']`.
- **Boot guard** — `Plugin::boot()` uses a static `$booted` flag + `$instance` check to prevent double-booting. Never call `new Plugin()` directly.
- **Register hooks in `Plugin::register_hooks()`** — all `add_action()` and `add_filter()` calls go here. Add new hooks at the landmark comment: `// --- Register additional feature hooks below this line ---`.
- **`add_option()` not `update_option()`** in `activate()` — `add_option()` is a no-op if the option exists, making reactivation safe.
- **Tabs, not spaces** — WordPress PHP standard uses tabs for indentation.

### JavaScript / CSS

- **WordPress ESLint + Stylelint** configs via `@wordpress/scripts`.
- **No inline JS or CSS** — all scripts and styles go through the webpack pipeline in `assets/src/` and are enqueued via `wp_enqueue_script()` / `wp_enqueue_style()`.
- **`strategy => 'defer'`** on all scripts — required by Plugin Check for non-blocking script loading.
- **Conditional loading** — assets load only on the plugin's own admin page (checked via hook suffix).

## Security Rules

Every PHP file that outputs data or handles user input must follow these rules.
Plugin Check and PHPCS enforce most of them, but AI assistants should apply them
from the first draft — not as a fixup pass.

### Output Escaping (escape at the point of echo, never earlier)

```php
// NEVER                              ALWAYS
echo $variable;                    // echo esc_html( $variable );
echo $url;                         // echo esc_url( $url );
echo $html_content;                // echo wp_kses_post( $html_content );
echo $attribute;                   // echo esc_attr( $attribute );
```

### Input Sanitization (sanitize + unslash on every input read)

```php
// NEVER                              ALWAYS
$_POST['field']                    // sanitize_text_field( wp_unslash( $_POST['field'] ) )
$_GET['id']                        // absint( $_GET['id'] )
```

`wp_unslash()` first because WordPress adds magic quotes to superglobals.

### Database Queries (always prepare, never interpolate)

```php
// NEVER
$wpdb->query( "DELETE FROM $table WHERE id = $id" );

// ALWAYS
$wpdb->query( $wpdb->prepare( "DELETE FROM {$table} WHERE id = %d", $id ) );
```

### Redirects (safe redirect + exit)

```php
// NEVER                              ALWAYS
wp_redirect( $url );               // wp_safe_redirect( $url );
                                   // exit;
```

### Other Security Rules

| NEVER | ALWAYS |
|-------|--------|
| `json_encode()` | `wp_json_encode()` |
| `move_uploaded_file()` | `wp_handle_upload()` |
| `extract()`, `eval()`, `create_function()`, backtick operator | Explicit variable assignment |
| `file_get_contents()` for URLs | `wp_remote_get()` |
| `curl_*` functions | `wp_remote_*()` family |
| Skip nonce checks | `wp_verify_nonce()` / `check_admin_referer()` on every state change |
| Skip capability checks | `current_user_can()` before any privileged action |

## PHPCS Suppression Patterns

Sometimes PHPCS rules must be suppressed for legitimate code (e.g., direct DB
queries with proper `$wpdb->prepare()`). Use the exact comment format below —
partial rule names won't suppress correctly.

```php
// Direct query with caching handled elsewhere:
// phpcs:ignore WordPress.DB.DirectDatabaseQuery.DirectQuery, WordPress.DB.DirectDatabaseQuery.NoCaching
$wpdb->query( $wpdb->prepare( "DELETE FROM {$table} WHERE id = %d", $id ) );

// Table name from $wpdb->prefix is safe, not user input:
// phpcs:ignore WordPress.DB.PreparedSQL.InterpolatedNotPrepared -- table name from $wpdb->prefix
$wpdb->get_results( $wpdb->prepare( "SELECT * FROM {$table} WHERE status = %s", $status ) );

// Schema changes during activation (dbDelta):
// phpcs:ignore WordPress.DB.DirectDatabaseQuery.SchemaChange
dbDelta( $sql );
```

Always add a trailing `--` comment explaining *why* the suppression is needed.

## Adding a New Feature

1. **Create the class** in `src/` following PSR-4 conventions.
   Example: `src/Cron/DailyReport.php` → namespace `StarterPlugin\Cron`.

2. **Add `declare( strict_types=1 );`** as the first statement (after `<?php` and docblock).

3. **Add the ABSPATH guard** after the namespace declaration:
   ```php
   defined( 'ABSPATH' ) || exit;
   ```

4. **Register hooks** in `Plugin::register_hooks()` at the landmark comment:
   ```php
   // --- Register additional feature hooks below this line ---
   $daily_report = new Cron\DailyReport();
   add_action( 'starter_plugin_daily_cleanup', [ $daily_report, 'run' ] );
   ```

5. **Write a test** in `tests/Unit/` mirroring the `src/` structure.
   Example: `tests/Unit/Cron/DailyReportTest.php`.

6. **Run checks** before committing:
   ```bash
   composer check   # PHPCS + PHPStan + PHPUnit
   npm run build    # Rebuild assets if changed
   ```

## Extension Skeletons

### Quick Reference

| Extension | Create file in | Register in `Plugin::register_hooks()` | Key gotcha |
|-----------|---------------|---------------------------------------|------------|
| REST endpoint | `src/Api/` | `add_action( 'rest_api_init', ... )` | Always set `permission_callback` (even `__return_true`) |
| Custom table | `src/Database/` | Check `{prefix}_db_version` on `plugins_loaded` | `require_once ABSPATH . 'wp-admin/includes/upgrade.php'` before `dbDelta()` |
| Cron job | `src/Cron/` or inline | `wp_schedule_event()` in `activate()` | Clean up in both `deactivate()` AND `uninstall.php` |
| Gutenberg block | Use `@wordpress/create-block` | N/A (block.json auto-registers) | `block.json` must have `"apiVersion": 3` |

Copy-paste these into new files. Each follows all conventions above.

### REST API Endpoint

```php
<?php
// File: src/Api/ExampleEndpoint.php

declare( strict_types=1 );

namespace StarterPlugin\Api;

defined( 'ABSPATH' ) || exit;

class ExampleEndpoint {

	private const ROUTE_NAMESPACE = 'starter-plugin/v1';

	public function register_routes(): void {
		register_rest_route(
			self::ROUTE_NAMESPACE,
			'/items',
			[
				'methods'             => 'GET',
				'callback'            => [ $this, 'get_items' ],
				'permission_callback' => [ $this, 'check_permission' ],
			]
		);
	}

	public function check_permission(): bool {
		return current_user_can( 'manage_options' );
	}

	public function get_items( \WP_REST_Request $request ): \WP_REST_Response {
		return new \WP_REST_Response( [ 'items' => [] ] );
	}
}

// Register in Plugin::register_hooks():
// add_action( 'rest_api_init', [ new Api\ExampleEndpoint(), 'register_routes' ] );
```

### Custom Database Table

```php
<?php
// File: src/Database/ExampleTable.php

declare( strict_types=1 );

namespace StarterPlugin\Database;

defined( 'ABSPATH' ) || exit;

class ExampleTable {

	public static function get_table_name(): string {
		global $wpdb;
		return $wpdb->prefix . 'starter_plugin_items';
	}

	public static function create_table(): void {
		global $wpdb;
		$table   = self::get_table_name();
		$charset = $wpdb->get_charset_collate();

		$sql = "CREATE TABLE {$table} (
			id bigint(20) unsigned NOT NULL AUTO_INCREMENT,
			title varchar(255) NOT NULL DEFAULT '',
			created_at datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
			PRIMARY KEY (id)
		) {$charset};";

		require_once ABSPATH . 'wp-admin/includes/upgrade.php';
		dbDelta( $sql );

		// update_option is correct here — db version must overwrite on each migration.
		// add_option is only for settings that should preserve user values on reactivation.
		update_option( 'starter_plugin_db_version', '1.0.0' );
	}
}

// In Plugin::activate():  ExampleTable::create_table();
// In Plugin::register_hooks(): check db version on plugins_loaded for migrations.
// In uninstall.php: $wpdb->query( "DROP TABLE IF EXISTS {$table}" );
```

### Cron Job

```php
<?php
// File: src/Cron/DailyCleanup.php

declare( strict_types=1 );

namespace StarterPlugin\Cron;

defined( 'ABSPATH' ) || exit;

class DailyCleanup {

	public const HOOK = 'starter_plugin_daily_cleanup';

	public static function schedule(): void {
		if ( ! wp_next_scheduled( self::HOOK ) ) {
			wp_schedule_event( time(), 'daily', self::HOOK );
		}
	}

	public static function unschedule(): void {
		wp_unschedule_hook( self::HOOK );
	}

	public function run(): void {
		// Your cleanup logic here.
	}
}

// In Plugin::activate():       Cron\DailyCleanup::schedule();
// In Plugin::deactivate():     Cron\DailyCleanup::unschedule();
// In Plugin::register_hooks(): add_action( DailyCleanup::HOOK, [ new Cron\DailyCleanup(), 'run' ] );
// In uninstall.php:            wp_unschedule_hook( 'starter_plugin_daily_cleanup' );
```

### Gutenberg Blocks

Use `@wordpress/create-block` to scaffold blocks — don't hand-roll them.
When adding a block, `block.json` **MUST** include `"apiVersion": 3` for
WP 7.0+ iframe editor compatibility. Plugin Check flags `apiVersion` < 3.

## Testing

### Running Tests

```bash
composer test          # PHPUnit only (Unit suite).
composer check         # PHPCS + PHPStan + PHPUnit (full CI check).
```

### Writing a Test

1. Create `tests/Unit/<Mirror>/YourClassTest.php` matching the `src/` structure.
2. Extend `StarterPlugin\Tests\TestCase` (provides Brain\Monkey setUp/tearDown).
3. Mock WordPress functions with Brain\Monkey:

```php
use Brain\Monkey\Functions;

Functions\expect( 'get_option' )
    ->once()
    ->with( 'starter_plugin_options', [] )
    ->andReturn( [ 'example_text' => 'hello' ] );
```

4. For hooks, use Brain\Monkey's hook expectations:

```php
Functions\expect( 'add_action' )
    ->once()
    ->with( 'admin_menu', \Mockery::type( 'array' ) );
```

5. Run `composer test` to verify.

### Key Testing Patterns

- **Brain\Monkey** mocks all WP functions — no WordPress installation needed.
- **Mockery assertions** are bridged to PHPUnit in the base `TestCase` class.
- **ABSPATH** is defined in `tests/bootstrap.php` so source files load correctly.
- **`$GLOBALS['wpdb']`** is stubbed with `$options` and `$prefix` properties.

## Forbidden Rules

These are non-negotiable constraints. Violating any of them will cause Plugin
Check failure, wp.org rejection, or security vulnerabilities.

1. **NEVER** output unescaped data. Every `echo`, `printf`, HTML attribute, and
   URL must use the appropriate `esc_*()` function at the point of output.

2. **NEVER** use `$_GET`, `$_POST`, `$_REQUEST`, or `$_SERVER` without
   `wp_unslash()` + `sanitize_text_field()` (or the appropriate sanitizer).

3. **NEVER** interpolate variables into SQL. Always use `$wpdb->prepare()`.

4. **NEVER** use `wp_redirect()`. Always use `wp_safe_redirect()` followed by `exit;`.

5. **NEVER** use `extract()`, `eval()`, `create_function()`, or backtick operators.

6. **NEVER** use anonymous closures as hook callbacks — they cannot be unhooked.

7. **NEVER** ship `vendor/` or `node_modules/` in the distribution zip.

8. **NEVER** use `json_encode()` — use `wp_json_encode()`.

9. **NEVER** use `file_get_contents()` for remote URLs — use `wp_remote_get()`.

10. **NEVER** include a custom update checker (wp.org Guideline 8 — plugins
    hosted on wp.org must use wp.org's update mechanism exclusively).

11. **NEVER** change `Tested up to` or `Requires at least` without explicit
    instruction — incorrect values cause wp.org listing issues.

12. **NEVER** omit `defined( 'ABSPATH' ) || exit;` from any PHP file — Plugin
    Check flags direct file access.

13. **NEVER** load scripts without `strategy => 'defer'` — Plugin Check flags
    render-blocking scripts.

14. **NEVER** enqueue assets globally — load them only on pages where needed.

15. **ALWAYS** prefix all options, hooks, functions, classes, and constants with
    `starter_plugin_` / `StarterPlugin` / `STARTER_PLUGIN_` as appropriate.

16. **ALWAYS** verify nonces on every state-changing action.

17. **ALWAYS** check `current_user_can()` before any privileged operation.

18. **ALWAYS** clean up all plugin data in `uninstall.php` (options, transients,
    cron hooks, custom tables).

19. **ALWAYS** use `add_option()` (not `update_option()`) in activation hooks to
    preserve existing settings on reactivation.

20. **ALWAYS** use `npm ci` (not `npm install`) in CI to ensure deterministic
    builds from the lockfile.
